**Location:** `examples/bookstore`

A multi-service demo showing the full contract-driven development pattern with `hoppity-contracts` and `hoppity-operations`. Two services (order-service and catalog-service) communicate through events, commands, and RPC, with a runner that drives the demo flow.

## Middleware Used

- `withCustomLogger` — custom logger injection
- `withOutboundExchange` — per-service fanout exchange for observability
- `withOperations` — typed handlers and broker methods from domain contracts

## Key Concepts

- **`defineDomain`** — Shared Zod-validated contracts are the single source of truth for message schemas, exchange names, queue names, and routing keys.
- **`buildServiceTopology`** — Each service declaratively states which contracts it publishes, subscribes to, sends, handles, calls, or responds to. The topology (exchanges, queues, bindings, publications, subscriptions) is generated automatically.
- **`withOperations`** — Typed `broker.publishEvent()`, `broker.sendCommand()`, and `broker.request()` methods. Handlers are declared with `onEvent`, `onCommand`, and `onRpc`.
- **Outbound exchange tapping** — `withOutboundExchange` creates a per-service fanout exchange. Other services or tools can bind to it to observe all outbound traffic without coupling.
- **Event vs. command vs. RPC** — The example demonstrates when to use each operation type: events for broadcasting, commands for directed work, RPC for synchronous queries.

## Project Structure

```
examples/bookstore/
├── bookstore-contracts/    # Shared domain contracts (Zod schemas)
├── order-service/          # Publishes events, handles commands, responds to RPC
├── catalog-service/        # Subscribes to events, sends commands, calls RPC
└── runner/                 # Orchestrates the demo flow
```

## Running It

```bash
cd examples/bookstore
pnpm --filter @bookstore/runner start
```

The runner starts both services, then drives a demo flow:

1. **Create & query** — Creates an order, queries its status via RPC
2. **Cancel & query** — Cancels the order, queries again to confirm the updated status

## What to Look For

- Contracts in `bookstore-contracts` define the schema once — both services import and use the same contract objects
- `buildServiceTopology` generates different topology for each service based on what it declares
- `withOperations` wires typed handlers and provides typed outbound methods
- The outbound exchange lets you observe all of a service's outbound traffic from RabbitMQ Management UI
- Zod validation catches schema mismatches at runtime with clear error messages
