**Location:** `examples/rpc`

Two-service example demonstrating request/response messaging over RabbitMQ. An initiator sends periodic RPC calls, a handler processes them and returns responses.

## Middleware Used

- `withCustomLogger` — custom logger injection
- `withBasicServiceComms` — custom middleware (inbound/outbound exchanges)
- `withRpcSupport` — RPC exchange, queues, bindings, and broker methods

## Key Concepts

- **`broker.request()`** — Promise-based RPC calls with automatic correlation ID tracking. Each call generates a unique correlation ID and returns a Promise that resolves when the response arrives.
- **`broker.addRpcListener()`** — Register handlers that receive the request payload and return a response value. The response is published back to the caller's reply queue.
- **Per-instance exclusive queues** — Each service instance gets its own reply queue and inbound queue, named using `instanceId`. This ensures responses route to the correct instance.
- **`BrokerWithExtensions`** — TypeScript utility type for combining multiple middleware extensions on a single broker type.

## Running It

```bash
cd examples/rpc
pnpm dev:both
```

The initiator sends RPC requests every few seconds. The handler processes each request and returns a response. Both sides log the full request/response cycle.

## Architecture

```
Initiator                          Handler
    │                                  │
    ├── request("getUser", payload) ──►│
    │   (via RPC topic exchange)       │
    │                                  ├── processes request
    │◄── response (via reply queue) ───┤
    │                                  │
```

Each service has:

- A **reply queue** (exclusive, per-instance) for receiving responses
- An **inbound queue** for receiving requests
- Bindings on the shared **RPC topic exchange** for routing

## What to Look For

- The RPC exchange in RabbitMQ Management UI shows topic-based routing
- Each instance creates its own exclusive reply queue
- Timeout handling — if the handler doesn't respond, the request Promise rejects with `RpcErrorCode.TIMEOUT`
- Correlation IDs tie requests to responses across the message bus
