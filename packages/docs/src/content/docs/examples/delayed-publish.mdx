**Location:** `examples/delayed-publish`

Two-service example demonstrating TTL-based delayed message delivery. A scheduler publishes messages with a configurable delay, a processor consumes them after the delay expires.

## Middleware Used

- `withCustomLogger` — custom logger injection
- `withBasicServiceComms` — custom middleware (inbound/outbound exchanges)
- `withDelayedPublish` — delayed publish with TTL-based wait/ready queue infrastructure
- `withSubscriptions` — auto-wired subscription handlers

## Key Concepts

- **Wait queue → dead-letter → ready queue** — Messages sit in the wait queue until their TTL expires. RabbitMQ then routes them via the dead-letter exchange to the ready queue, where the middleware re-publishes them to the original destination.
- **`broker.delayedPublish()`** — Publish a message that will be delivered after a specified delay.
- **Custom middleware** — `withBasicServiceComms` shows how to write your own middleware that adds inbound/outbound exchanges to the topology.

## Running It

```bash
cd examples/delayed-publish
pnpm dev:both
```

The scheduler publishes delayed messages on an interval. The processor receives them after the delay expires and logs the result.

## What to Look For

- The wait queue appears in RabbitMQ Management UI with a TTL policy
- Messages move from wait → dead-letter → ready automatically
- The processor receives messages only after the delay has elapsed
- Re-publish retries handle transient failures transparently
