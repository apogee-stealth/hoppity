---
title: hoppity-delayed-publish
description: TTL-based delayed message publishing with automatic retry.
---

TTL-based delayed message publishing for hoppity brokers. Uses RabbitMQ dead-letter exchanges to hold messages in a wait queue until their TTL expires, then routes them to a ready queue for re-publication to the original destination.

## Installation

```bash
npm install @apogeelabs/hoppity-delayed-publish
```

:::note
Requires `@apogeelabs/hoppity` and `rascal@^20.1.1` as peer dependencies.
:::

## Usage

```typescript
import hoppity from "@apogeelabs/hoppity";
import { withDelayedPublish, type DelayedPublishBroker } from "@apogeelabs/hoppity-delayed-publish";
import { randomUUID } from "crypto";

const broker = (await hoppity
    .withTopology(topology)
    .use(
        withDelayedPublish({
            serviceName: "scheduler",
            instanceId: randomUUID(),
        })
    )
    .build()) as DelayedPublishBroker;

// Publish with a 5-second delay
await broker.delayedPublish("my_publication", payload, undefined, 5_000);
```

## Configuration

`withDelayedPublish` accepts a `DelayedPublishOptions` object:

| Option         | Type      | Required | Default | Description                                       |
| -------------- | --------- | -------- | ------- | ------------------------------------------------- |
| `serviceName`  | `string`  | Yes      | —       | Names the wait/ready queues and exchanges         |
| `instanceId`   | `string`  | Yes      | —       | Unique instance identifier                        |
| `defaultDelay` | `number`  | No       | `5000`  | Default delay in milliseconds                     |
| `maxRetries`   | `number`  | No       | `3`     | Maximum re-publish retry attempts                 |
| `retryDelay`   | `number`  | No       | `1000`  | Delay between retries (ms)                        |
| `durable`      | `boolean` | No       | `true`  | Whether wait/ready queues survive broker restarts |

## How It Works

The middleware creates a two-queue pipeline per service:

1. **Wait queue** — Messages are published here with a TTL matching the desired delay. The queue has a dead-letter exchange configured.
2. **Dead-letter exchange** — When TTL expires, RabbitMQ routes the message here automatically.
3. **Ready queue** — Consumes from the dead-letter exchange. The `onBrokerCreated` callback subscribes to this queue and re-publishes the original message to its intended destination.

If re-publication fails, the middleware retries up to `maxRetries` times with `retryDelay` between attempts.

## Error Codes

The `DelayedPublishErrorCode` enum covers failure modes:

- `QUEUE_FULL` — Wait queue rejected the message
- `REPUBLISH_FAILED` — Re-publication to the original destination failed
- `MAX_RETRIES_EXCEEDED` — All retry attempts exhausted
- `INVALID_DELAY` — Delay value is not a positive number
