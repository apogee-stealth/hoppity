Domain-centric messaging contracts for hoppity topology generation. Define events, commands, and RPC operations with Zod schemas, then derive deterministic RabbitMQ topology — exchanges, queues, bindings, publications, and subscriptions — from those contracts.

## Installation

```bash
npm install @apogeelabs/hoppity-contracts zod
```

## Defining a Domain

A domain groups related events, commands, and RPC operations under a single namespace:

```typescript
import { defineDomain } from "@apogeelabs/hoppity-contracts";
import { z } from "zod";

const Order = defineDomain("order", {
    events: {
        placed: z.object({ orderId: z.string(), total: z.number() }),
        cancelled: z.object({ orderId: z.string(), reason: z.string() }),
    },
    commands: {
        fulfill: z.object({ orderId: z.string(), warehouseId: z.string() }),
    },
    rpc: {
        getStatus: {
            request: z.object({ orderId: z.string() }),
            response: z.object({ status: z.string(), updatedAt: z.string() }),
        },
    },
});
```

Each contract object (`Order.events.placed`, `Order.commands.fulfill`, `Order.rpc.getStatus`) carries:

- The Zod schema for validation
- Deterministic naming for exchanges, queues, bindings, publications, and subscriptions
- Full TypeScript type inference — no `as` casts needed

## Building Service Topology

`buildServiceTopology` generates RabbitMQ topology from contract declarations:

```typescript
import { buildServiceTopology } from "@apogeelabs/hoppity-contracts";

const topology = buildServiceTopology(baseConfig, "order-service", t => {
    t.publishesEvent(Order.events.placed);
    t.publishesEvent(Order.events.cancelled);
    t.handlesCommand(Order.commands.fulfill);
    t.respondsToRpc(Order.rpc.getStatus);
});
```

The `TopologyBuilder` callback provides these declaration methods:

| Method              | Description                                |
| ------------------- | ------------------------------------------ |
| `publishesEvent`    | Declares the service publishes this event  |
| `subscribesToEvent` | Subscribes to an event from another domain |
| `sendsCommand`      | Declares the service sends this command    |
| `handlesCommand`    | Handles a command (creates inbound queue)  |
| `callsRpc`          | Makes RPC calls to another service         |
| `respondsToRpc`     | Handles RPC requests and returns responses |

Each declaration generates the appropriate exchanges, queues, bindings, publications, and subscriptions in the Rascal topology. Names are deterministic — derived from the domain name, operation name, and service name.

## Outbound Exchange

`withOutboundExchange` is a middleware that creates a per-service fanout exchange. Other services can bind to this exchange to observe all outbound traffic without coupling:

```typescript
import { withOutboundExchange } from "@apogeelabs/hoppity-contracts";

const broker = await hoppity
    .withTopology(topology)
    .use(withOutboundExchange("order-service"))
    .build();
```

## Handler Options

When subscribing to events, handling commands, or responding to RPCs, you can customize queue behavior:

```typescript
t.handlesCommand(Order.commands.fulfill, {
    queueType: "quorum",
    redeliveries: { limit: 3, counter: "shared" },
    deadLetter: true,
});
```

| Option         | Type      | Description                                    |
| -------------- | --------- | ---------------------------------------------- |
| `queueType`    | `string`  | Queue type (`"classic"` or `"quorum"`)         |
| `redeliveries` | `object`  | Rascal redelivery configuration                |
| `deadLetter`   | `boolean` | Enable dead-letter routing for failed messages |

## Contracts + Operations

Contracts define the _what_ — operations handle the _how_. See [hoppity-operations](/hoppity/packages/hoppity-operations/) for wiring typed handlers and outbound methods onto the broker.
