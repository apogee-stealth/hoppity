title: hoppity-subscriptions
description: Validate and auto-wire RabbitMQ subscription handlers to a Rascal broker.
---

Validates and auto-wires RabbitMQ subscription handlers to a Rascal broker. Ensures every handler key maps to a real subscription in the topology (fail-fast), then attaches message, error, and invalid-content listeners during the `onBrokerCreated` callback phase.

:::tip
This should be the **last** middleware in the pipeline so it validates against the fully-assembled topology.
:::

## Installation

```bash
npm install @apogeelabs/hoppity-subscriptions
```

:::note
Requires `@apogeelabs/hoppity` and `rascal@^20.1.1` as peer dependencies.
:::

## Usage

```typescript
import hoppity from "@apogeelabs/hoppity";
import { withSubscriptions } from "@apogeelabs/hoppity-subscriptions";

const broker = await hoppity
    .withTopology(topology)
    .use(
        withSubscriptions({
            on_order_placed: async (message, content, ackOrNack) => {
                console.log("Order placed:", content);
                ackOrNack();
            },
            on_payment_received: async (message, content, ackOrNack) => {
                console.log("Payment:", content);
                ackOrNack();
            },
        })
    )
    .build();
```

The handler keys (`on_order_placed`, `on_payment_received`) must match subscription names defined in the Rascal topology. If they don't, the middleware throws immediately — no silent failures.

## Handler Signature

Each handler receives the same arguments as a Rascal subscription handler:

```typescript
type SubscriptionHandler = (
    message: object,
    content: any,
    ackOrNackFn: (err?: Error, options?: object) => void,
    broker: BrokerAsPromised
) => Promise<void> | void;
```

## Validation

You can validate handlers independently of the middleware pipeline:

```typescript
import { validateSubscriptionHandlers } from "@apogeelabs/hoppity-subscriptions";

const result = validateSubscriptionHandlers(topology, handlers);

if (!result.isValid) {
    console.log("Missing:", result.missingSubscriptions);
    console.log("Invalid:", result.invalidHandlers);
    console.log("Available:", result.availableSubscriptions);
}
```

The `ValidationResult` tells you exactly what's wrong — which handler keys don't match any subscription, and which subscriptions exist in the topology.

## Two-Phase Execution

1. **Middleware phase** — Validates handler keys against the topology. Throws if any key doesn't map to a subscription.
2. **onBrokerCreated phase** — Calls `broker.subscribe()` for each handler and wires up message, error, and invalid-content listeners.
