title: hoppity-rpc
description: Request/response RPC over RabbitMQ with correlation IDs and per-instance queues.
---

Hoppity middleware for request/response RPC over RabbitMQ. Adds `.request()`, `.addRpcListener()`, and `.cancelRequest()` to a hoppity broker using correlation IDs, per-instance exclusive queues, and a shared topic exchange.

## Installation

```bash
npm install @apogeelabs/hoppity-rpc
```

:::note
Requires `@apogeelabs/hoppity` and `rascal@^20.1.1` as peer dependencies.
:::

## Usage

```typescript
import hoppity from "@apogeelabs/hoppity";
import { withRpcSupport, type RpcBroker } from "@apogeelabs/hoppity-rpc";
import { randomUUID } from "crypto";

const broker = (await hoppity
    .withTopology(topology)
    .use(
        withRpcSupport({
            serviceName: "my-service",
            instanceId: randomUUID(),
        })
    )
    .build()) as RpcBroker;
```

### Making RPC Calls

```typescript
const response = await broker.request("getUser", { userId: "123" });
```

### Handling RPC Requests

```typescript
broker.addRpcListener("getUser", async request => {
    const user = await db.findUser(request.payload.userId);
    return { payload: user };
});
```

### Cancelling Requests

```typescript
const correlationId = "some-correlation-id";
broker.cancelRequest(correlationId);
```

## Configuration

`withRpcSupport` accepts an `RpcMiddlewareOptions` object:

| Option           | Type     | Required | Default | Description                                      |
| ---------------- | -------- | -------- | ------- | ------------------------------------------------ |
| `serviceName`    | `string` | Yes      | —       | Used to name the RPC exchange and queues         |
| `instanceId`     | `string` | Yes      | —       | Unique instance ID for per-instance reply queues |
| `rpcExchange`    | `string` | No       | `"rpc"` | Name of the shared RPC topic exchange            |
| `defaultTimeout` | `number` | No       | `30000` | Default timeout (ms) for RPC requests            |

## How It Works

1. **Topology phase** — Creates a shared topic exchange, a per-instance exclusive reply queue, and bindings for routing responses back to the caller.
2. **Broker phase** — Patches `request()`, `addRpcListener()`, and `cancelRequest()` onto the broker in the `onBrokerCreated` callback.
3. **Request flow** — Each `.request()` generates a correlation ID, publishes to the RPC exchange with a reply-to header, and returns a Promise that resolves when the response arrives (or rejects on timeout).

## Error Codes

The `RpcErrorCode` enum covers failure modes:

- `TIMEOUT` — Request exceeded the timeout
- `METHOD_NOT_FOUND` — No handler registered for the RPC name
- `HANDLER_ERROR` — Handler threw an exception
- `CANCELLED` — Request was explicitly cancelled
- `SERVICE_UNAVAILABLE` — Target service is unreachable
